# Design Skills

Claude Code skills for making web apps that feel considered. Grounded in Apple's Human Interface Guidelines, adapted for the web.

The idea is simple: you have a `design.md` that holds your entire design system — colors, type, spacing, components, everything. When you ask a design question, Claude gives you a recommendation rooted in Apple HIG and proposes changes back to `design.md`. Over time, that file becomes a living record of every design decision you've made, and your system stays consistent even as it grows.

## Getting Started

**If you don't have a `design.md` yet**, start here:

```
/hig-init
```

It'll ask a few questions about your product, look at your codebase, and generate a complete `design.md` with HIG-aligned tokens and specs. That file is the foundation everything else builds on.

**If you already have a `design.md`**, just start asking questions:

```
/hig should this button be filled or outlined?
```

You'll get a recommendation, and if it means your design system should change, Claude will propose the update. You approve it, `design.md` evolves, and the next question you ask already knows about that decision.

## Skills

| | |
|---|---|
| `/hig-init` | Generate your `design.md` from scratch. Start here. |
| `/hig` | Ask any design question. Routes to the right module, or audits your whole system. |
| `/hig-foundations` | Color, typography, spacing, layout, motion, accessibility, dark mode, input methods. |
| `/hig-patterns` | Navigation, modals, onboarding, loading, feedback, forms, sharing, notifications. |
| `/hig-components` | Buttons, menus, inputs, lists, alerts, popovers, sheets, tabs, charts, and more. |

You can go straight to a specific module if you know what you need, or just use `/hig` and let it figure out the routing.

## Setup

Drop the skill folders into your project:

```
your-project/
  hig/
  hig-init/
  hig-foundations/
  hig-patterns/
  hig-components/
  design.md          <-- generated by /hig-init
```

That's it.

## Examples

```
/hig-init
/hig should this card have a border or a shadow?
/hig audit my design system
/hig-foundations review my color tokens for dark mode
/hig-patterns how should onboarding work for first-time users?
/hig-components review my alert component
```

## Why design.md?

Design decisions tend to get lost — scattered across conversations, Figma comments, PR reviews, your memory. `design.md` gives them a home. Every recommendation you accept gets written back, so the system stays coherent even when you're making changes weeks apart. It's the source of truth that Claude reads before every answer, which means your 50th design question is informed by the first 49.
